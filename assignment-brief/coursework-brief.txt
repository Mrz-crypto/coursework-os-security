ST5039CMD Programming and Operating Systems CW1 - Practical Summary

This is a summary of the supplied coursework PDF. It is kept short so the
repository documents what the code is trying to demonstrate.

Overall requirements

- Implement the practical work in C on a Unix/Linux system.
- Upload source code, diagrams, scripts, logs, and supporting files to a
  public GitHub repository.
- Write a report that explains the implementation, observations, evidence,
  limitations, and references.
- Keep the work academic and personally written.

Task 1 - Privilege separated password validation

Weighting: 40 percent, around 2000 report words.

Required practical work:

- Write two independent C programs: Frontend.c and Backend.c.
- Use separate operating system processes, not threads.
- Use local IPC, especially a UNIX domain socket.
- Pass authentication requests from the frontend to the backend.
- Backend must validate requests and reject malformed or invalid requests.
- Demonstrate privilege dropping with an actual setresuid() call.
- Show a runtime privilege check using geteuid(), getresuid(), or /proc.
- Handle password memory carefully and clear sensitive buffers explicitly.

Main investigation themes:

- Why one process handling both user input and sensitive validation is risky.
- How least privilege is enforced with multiple processes.
- What can go wrong if privilege dropping is implemented incorrectly.
- How independent processes exchange requests securely.
- Why UNIX domain sockets are suitable for local privileged communication.
- How operating systems allow controlled data sharing without breaking
  isolation.
- Why permanently dropping root privileges should be irreversible.
- How to verify privilege dropping through runtime observations.
- How retained elevated privileges could benefit an attacker.
- Why clearing passwords from memory matters.
- Why normal memory clearing may be optimized away.
- How explicit memory clearing reduces that risk.

Task 2 - User-space malware analysis sandbox

Weighting: 50 percent, around 3000 report words.

Required practical work:

- Write Sandbox.c.
- Provide test binaries.
- Use fork() to create a child process.
- Use execve() to run the untrusted program.
- Keep monitoring and enforcement in the parent process.
- Monitor execution time and resource use externally.
- Use signals such as SIGTERM and SIGKILL to terminate unsafe children.
- Use multiple pthread monitor threads in the supervising process.
- Keep shared monitor state race-free, using atomics or locks.
- Produce logs showing normal execution and forced termination.

Main investigation themes:

- Why process-level isolation is better than in-process containment for
  untrusted code.
- How the parent-child process model supports monitoring and enforcement.
- What changes when untrusted code runs through execve().
- How CPU time and wall-clock time can be measured externally.
- Why time enforcement must not depend on the child process cooperating.
- What role signals play in termination policies.
- Why concurrency is useful in a sandbox monitor.
- How shared state between monitor threads can create race conditions.
- Why C synchronization should use atomics rather than volatile variables.
- How timing channels and side effects may bypass simple sandboxes.
- Why these limitations matter in real malware analysis.

Report requirements

- Include practical evidence, such as terminal logs and screenshots.
- Include diagrams explaining Task 1 and Task 2.
- Include academic references, websites, books, and man pages used.
- Use a consistent referencing style.
- Present the report clearly with good grammar and formatting.
