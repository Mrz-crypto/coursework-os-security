TASK 1:(40%) - 2000 words
" Privilege separation in Password validation (Process Isolation and 
Secure Design)"
In this task, you will develop and test a privilege separated authentication 
service that distributes responsibility across various operating system 
processess. One process accepts user input, while another executes crucial 
validation procedures before permanently relinquishing enchanced privileges. 
Your solution must be written in C program and based on operating system 
techniques rather than application level abstractions. 
The objective of this task is to investigate how process isolation, inter 
process communicatoin, privilege management, and secure memory 
handling can be used to implement the principle of least privilege on a 
Unix based operating system.
Write at least two C programs:
1. Frontend.c
2. Backend.c
Use:
1. Process Isolation - two distinct executables eg, fork(), execve()
2. Secure IPC - credentials passed via UNIX domain socket, syscalls
3. Privilege dropping ( setuid / setresuid ) - Backend  calls, Runtime check
4. Attack resistance - Backend rejecting requests, if not a validation 
5. Shared memory for safety - Password buffer, Disassembly
If there is no working program, the task is incomplete.
Deliverables:
1. Two independent processess, not thread.
2. Actual setresuid( ) call
3. Runtime check using geteuid( ) or / proc
Investigation Questions:
Your are also required to document your analysis and verification that 
would answer the following questions:
1. 
why is it considered insecure for a single process to receive user input and access sensiteve 
authencition data on a Linux system?
2. How can the principle of least privilee be enforced at the operating system level using multiple 
processess rather than threads?
3. What risks remain if privilege dropping is incorrectly implemented?
4. How can two independent processess securely exchange authentication requests and results 
without exposing sensitive data to unauthorized processess?
5. Why are UNIX domain sockets preferable to network sockets for local privileged 
communication?
6. What operating system mechanisms allow controlled sharing of data between processess 
without violating isolation?
7. How can a process permanently relinquish root privileges after startup, and why must this be 
irreversible?
8. What observation system indicators can be used to verify that privilege dropping has 
succeeded?
9. How would an attacker benefit if a process unintentionally retained elevated privileges?
10. Why is clearing sensitive data from memory as security requirement even after authentication 
is complete?
11. Why can standard memory clearing functions be unreliable according to the C language 
memory model?
12. How does the use of explicit memory primitives mitigate this risk?
TASK 2: (50%) - 3000 Words
" User Space Malware Analysis Sandbox (Process Control, Resource 
Isolation and Concurrency ) "
Security researchers routinely use untrusted binaries to watch behavior and 
assess risk. Running such programs directly on a host system without isolation 
may result in system compromise, resource depletion, or data exfiltration. 
This work requires you to design and build a user space sandbox that runs 
untrusted application under stringent monitoring. The sandbox must rely on 
operating system process management and concurrency capabilities, rather 
than theoritical debate or signature based malware detection. This work 
focuses on how the operating system imposes control, rather than malware 
classificaton. 
The objective of this task is to investigaet how parent child process 
isolation, external resource monitoring, signals, and concurrency can be 
combined to limit the behavior of untrusted executables.
Implementation:
You must develop a sandbox controller using C programming that:
1. Fork( ) creates a child process, and execve( ) executes an unsafe binary. 
2. Supervises the child process after the parent process.
3. Monitors execution time and resource utilization from the outside. 
4. Uses operating system signals to enforce termination policies. 
5. To execute concurrent monitoring activities, the supervising process makes use of several 
threads (pthreads)
The untrusted binary must not take part in its own monitorin or 
termination.
Analysis and Observation:
1. Why is parent child isolation? Parent terminates child 
2. Why race free shared state?
3. Why forced termination?
Deliverables:
1. Sandbox.c
2. Test Binaries
3. Logs showing termination
Investigation Questions:
You are required to investigate and demonstrate with reference to your 
implementation and execution evidience following the question below:
1. Why is process level isolation preferred over in process containment for executing untrusted 
binaries?
2. How does the parent-child process model support monitoring and enforcement of execution 
policies?
3. What security assumptions change when untrusted code is executed via execve( )?
4. How can CPU usage and execution time be measured externally by supervising process?
5. Why must time based enforcement be independent of the monitored process's cooperation?
6. What role do signals play in enforcing termination policies?
7. Why is concurrency required in a sandbox monitoring system?
8. How can shared state between monitoring threads introduce subtle race condition?
9. Why does the C language memory model require atomic operations rather than volatile 
variables for synchronizations?
10. How could timing channels or side effects be used to bypass sandbox restrictions?
11. Why is understanding these limitations critical in real world malware analysis?
